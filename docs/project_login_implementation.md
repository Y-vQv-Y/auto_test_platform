# 在自动化测试平台中实现网站登录

## 1. 理解登录态管理的核心挑战

在自动化测试中，处理网站登录是一个常见的挑战。主要原因在于：

*   **安全性**：现代网站广泛使用 `HttpOnly` Cookie、验证码（如滑块、图片验证码）、二次认证等机制来增强安全性，防止自动化脚本直接模拟登录。
*   **登录态生命周期**：登录态（Session/Cookie）具有有效期，过期后需要重新登录，这要求测试平台具备会话管理和刷新能力。
*   **测试隔离**：每个测试用例或测试运行可能需要独立的登录会话，以避免相互干扰。

您之前尝试通过浏览器控制台的 JavaScript 脚本提取 Cookie 遇到 `undefined` 的问题，这正是 `HttpOnly` 安全机制的体现。`document.cookie` 无法访问 `HttpOnly` 标记的 Cookie，而这些 Cookie 往往是维持登录态的关键。因此，**手动通过 JavaScript 提取 Cookie 并非在本项目中实现登录的推荐方式**。

## 2. 本项目推荐的登录实现方式

本自动化测试平台已内置一套专门用于处理复杂登录场景的机制，旨在解决 `HttpOnly` 限制和验证码问题，并提供登录态的自动管理。您应该充分利用这些内置功能，而不是尝试手动提取和注入 Cookie。

核心机制包括：

1.  **手动滑块验证码处理**：对于需要人工交互的验证码（如滑块验证），平台会启动一个真实的浏览器窗口，让用户手动完成验证，然后自动捕获并保存登录态。
2.  **登录态自动保存与加载**：成功登录后，平台会将所有必要的会话信息（包括 Cookie 和 LocalStorage）加密存储在数据库中，并在后续测试运行中自动加载。
3.  **登录态有效性校验**：在每次测试运行前，平台会主动校验已保存的登录态是否仍然有效，避免使用过期会话进行测试。
4.  **重新登录引导**：如果登录态失效，平台会提示用户重新进行登录操作。

## 3. 在项目中配置和使用登录功能

以下是在您的自动化测试平台中实现网站登录的详细步骤：

### 3.1 配置项目部署 URL

首先，确保您的项目配置了正确的部署 URL，这是平台进行登录和测试的基础。

1.  登录平台，进入您的项目详情页。
2.  点击项目名称旁边的“编辑”按钮（铅笔图标）。
3.  在弹出的编辑窗口中，填写 **“部署 URL”** 字段，例如 `https://your-website.com/login` 或 `https://your-website.com`。
4.  保存项目配置。

### 3.2 处理首次登录或登录态失效

当您第一次为项目配置登录，或者已保存的登录态失效时，您需要通过以下方式进行登录操作：

1.  **进入项目详情页**：在项目列表中点击您的项目名称。
2.  **查看登录状态卡片**：在项目详情页的顶部，您会看到一个“登录状态”卡片。如果显示“未登录”或“已登录 (失效)”，则需要进行登录操作。
3.  **点击“重新登录”按钮**：
    *   点击登录状态卡片中的“重新登录”按钮（手机图标）。
    *   平台会启动一个 Playwright 浏览器实例，并导航到您配置的部署 URL。
    *   **重要**：此时，您需要**手动**在弹出的浏览器窗口中完成登录过程，包括输入用户名、密码、以及处理任何验证码（如滑块验证）。
    *   一旦您在弹出的浏览器中成功登录并跳转到登录后的页面，平台会自动捕获当前的 Cookie 和 LocalStorage，并将其保存到数据库中。
    *   浏览器窗口会自动关闭，平台会提示“重新登录成功”。
4.  **手动粘贴 Cookie (备用)**：
    *   如果平台无法自动捕获（例如，某些特殊验证码导致浏览器无法自动关闭），或者您更倾向于手动控制，可以在“重新登录”按钮旁边的“处理验证码登录”弹窗中，手动粘贴您从浏览器开发者工具中获取的 Cookie JSON 字符串。
    *   **注意**：如前所述，手动粘贴的 Cookie 可能不包含 `HttpOnly` Cookie，这可能导致登录不完整。因此，**优先使用“重新登录”按钮让平台自动处理**。

### 3.3 自动校验与测试运行

一旦您成功保存了登录态，后续的测试运行将自动利用这些信息：

1.  **测试运行前自动校验**：
    *   当您创建新的测试运行并选择执行测试用例时，平台会在实际执行测试用例之前，自动调用 `check_session_validity` 方法。
    *   该方法会使用已保存的 Cookie 和 LocalStorage 启动一个轻量级的 Playwright 实例，访问您的部署 URL，并检查页面上是否存在登录成功的标志（如用户头像、欢迎信息等）。
    *   如果校验通过，测试将继续执行。
    *   如果校验失败（登录态已过期），平台会记录警告，并可能导致测试用例因未登录而失败。此时，您需要回到项目详情页，点击“重新登录”按钮刷新登录态。
2.  **测试用例中无需再次登录**：
    *   在您编写的测试用例代码中，**无需**再包含 `page.goto('/login')` 或输入用户名密码的步骤。
    *   平台会在每个测试用例执行前，自动将已保存的登录态注入到 Playwright 的 `page.context` 中，确保测试用例在已登录的状态下运行。
    *   您的测试用例可以直接从登录后的页面开始操作。

## 4. 最佳实践

*   **定期检查登录态**：虽然平台会进行自动校验，但建议您在关键测试运行前，手动点击“校验登录态”按钮，确保会话处于最新有效状态。
*   **使用非无头模式处理登录**：在处理验证码或首次登录时，确保平台以非无头模式（`headless=False`）启动浏览器，以便您进行人工交互。
*   **避免在测试用例中硬编码登录逻辑**：将登录逻辑交给平台管理，使测试用例更专注于业务流程的验证，提高用例的复用性和可维护性。

通过遵循上述步骤，您将能够在本自动化测试平台中高效、可靠地管理网站登录，确保您的测试用例始终在正确的会话环境下运行。
